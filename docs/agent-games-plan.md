# План: каркас экспериментов «эмерджентная репутация в популяции LLM-агентов»

Дата составления: 2026-05-28

## 1. Цель и исследовательский вопрос

Построить параметризуемую обёртку для экспериментов, в которых популяция
ИИ-агентов раунд за раундом играет в коммуникативную игру. Первая игра —
повторяющаяся дилемма заключённого с cheap talk. Главный исследовательский
вопрос: **возникнет ли спонтанно передача информации об агентах третьим лицам
(gossip) → институт репутации** (indirect reciprocity).

Каркас общий (игра — плагин), но проверяется на одной реальной игре.

## 2. Зафиксированный контракт (принятые решения)

| Параметр | Решение |
|---|---|
| Базовая игра | Тайный выбор числа 0–9 + cheap talk; равно=CC, +1 (mod 10)=обман (DC/CD), иначе=DD. Платежи PD (T>R>P>S) |
| Популяция | N агентов с устойчивыми ID; матчинг — pluggable (`random` / `scheduled` / `choice`) |
| Окончание | По числу раундов R **или** пользовательский останов (с graceful-флашем) |
| Эволюция | Опц. эпохи: между эпохами K слабейших гибнут, K сильнейших дают потомков (копия `AgentSetup`, свежий ID, пустая память). По умолчанию выкл (`num_epochs`=1) |
| Тип | Соревновательная: каждый максимизирует свой кумулятивный платёж |
| Обман | Разрешён: можно нарушить договорённость о кооперации; можно врать о третьих |
| Канал | Свободный естественный язык, без ограничений (кроме тех. max_tokens) |
| Память | Двухъярусная: контекстное окно W раундов сырьём + self-notes раз в N раундов; оба — параметры свипа |
| Модели | Единый OpenAI-совместимый клиент (`base_url`+`model` по конфигу): локальная Ollama / Cerebras / Gemini / OpenAI / Anthropic; разным агентам — разные модели |
| Выход | Транскрипты + авто-метрики на прогон |
| Метрика репутации | **Отложена**: транскрипты пишем в форме, удобной для анализа; оставляем чистый хук |
| Масштаб | Неизвестен → закладываем возобновляемость, параллелизм пар, оценщик стоимости |
| Стек | Python, async |

## 3. Архитектура (слои)

Пять независимых слоёв:

1. **Провайдеры LLM** — единый интерфейс к моделям (Anthropic/OpenAI), ретраи,
   учёт токенов, кэширование статичного префикса.
2. **Агент** — единый `act(phase)`: игра, матчинг и память дёргают агента
   одинаково (фазы `TALK`, `DECIDE`, `CHOOSE_PARTNER`, `CONSENT`, `REFLECT`,
   `NOTES`), всё логируется единообразно.
3. **Игра (`Game`)** — pluggable-логика одного парного взаимодействия (play-фаза):
   cheap talk + решение + платежи + что попадает в память. PD — первая реализация.
4. **`Matchmaker` (планировщик раунда)** — pluggable-логика «кто с кем играет».
   Может быть неинтерактивным (random / scheduled) или интерактивным (choice —
   спрашивает агентов про выбор партнёра и согласие). Выдаёт `RoundPlan`.
5. **Оркестрация + свип + хранилище** — цикл раундов (матчинг → игра), прокладка
   памяти, чекпоинты; генерация конфигов из сетки, повторы/сиды, запись
   артефактов, индекс, возобновляемость.

Главный архитектурный шов — разделение **`Matchmaker`** (кто играет) и **`Game`**
(как играет пара). Схема матчинга — основной экспериментальный рычаг (§9).

## 4. Модель домена

- **AgentSetup** («генотип»): provider, model, temperature, persona/system-prompt.
  Неизменяемый; именно он копируется при размножении (§8).
- **Agent** (живой экземпляр): `AgentSetup` + `id` (устойчивый, напр. `A01`) +
  накопленная память/notes + очки эпохи (обнуляются в начале каждой эпохи;
  пожизненная сумма восстановима из записей для анализа). Создаётся из сетапа.
- **Population** (ростер): рантайм-структура — мутабельный список живых `Agent` с
  устойчивыми, не переиспользуемыми ID. Мутаторы `add(setup)` (свежий ID, пустая
  память, score=0) / `remove(id)` / `next_id()` — через них `SelectionPolicy`
  рождает потомков и выбраковывает слабейших между эпохами (§8).
- **PopulationGenerator** (pluggable, как `Matchmaker`): стратегия «как собрать
  стартовый состав» — `build(rng) -> Population`, выбирается по `population.kind`
  (свип-ось, §9). Реализации: `roster` (явный список персон/провайдеров,
  циклически до N) / `homogeneous` / `mixed`. Живёт в пакете `population/` (§14).
  В плоском прогоне (`num_epochs=1`) состав не меняется после генерации.
- **Memory**: список записей о прошлых раундах (с кем играл, полный cheap-talk,
  собственное решение + приватное обоснование, действие партнёра, платёж).
  Представление памяти — параметр (см. §6).
- **Phase**: что агента просят сделать сейчас — `kind` (`TALK` / `DECIDE` /
  `CHOOSE_PARTNER` / `CONSENT` / `REFLECT` / `NOTES`), отрендеренный `context`
  (правила + память + ситуация + опции) и схема ожидаемого структурированного
  вывода. Через `Phase` игра, матчинг и память обращаются к агенту единообразно.
- **Game**: интерфейс — `play_pairing(agent_a, agent_b, round, cfg, rng) ->
  PairingRecord`; задаёт play-фазу (cheap talk + решение), платежи и что именно
  агент запоминает.
- **PairingRecord**: ID пары, раунд, полный транскрипт cheap talk (реплики с
  автором, публичным текстом и флагом `ready`), приватные обоснования каждого,
  выбранные числа (0–9), исход, платежи, приватная пост-игровая рефлексия каждого
  (если `reflect_after` включает пару), расход токенов, тайминги.
- **Matchmaker**: интерфейс — `setup(population, rng, cfg)`,
  `async plan_round(population, round_idx, actor) -> RoundPlan`,
  `disclosure_for(agent, round_idx) -> str|None`. `actor` — колбэк вызова агента
  (чтобы не дублировать LLM-плумбинг). Реализации: `random` / `scheduled` /
  `choice` (§9).
- **RoundPlan**: `pairings` (1..N/2 пар), `idle` (агенты без игры в этом раунде),
  `events` (артефакты выбора/отказа из интерактивного матчинга).
- **Round**: `RoundPlan` (матчинг + события) + список `PairingRecord` по сыгранным
  парам + платежи `idle`-агентам (`idle_payoff`).
- **Epoch**: блок `epoch_length` раундов с фиксированным ростером; между эпохами —
  отбор. `SelectionPolicy.select(population, epoch_stats, rng, cfg) ->
  PopulationDelta` (кого удалить, кого добавить).
- **PopulationDelta**: K удаляемых (слабейшие) + K добавляемых потомков (новые ID
  из `AgentSetup` родителей, пустая память); фиксирует родословную (child→parent).
- **Episode (прогон)**: одна популяция, прогнанная `num_epochs`×`epoch_length`
  раундов (с отбором между эпохами) под фиксированный конфиг + сид. Единица свипа.
- **Experiment**: свип — множество эпизодов по сетке параметров × повторы.

## 5. Базовая игра: «число 0–9» (PD-эквивалент) + cheap talk (детально)

Вместо абстрактного C/D агенты тайно выбирают число 0–9; исход пары определяется
соотношением чисел и отображается на платежи PD.

Протокол одного парного взаимодействия:

1. **Сетап.** Каждому агенту сообщаются: его ID, ID партнёра, номер раунда,
   правила и матрица платежей. (Знание полного ростера ID — параметр, §16.)
2. **Cheap talk.** Поочерёдные реплики свободным текстом: договариваются, какое
   число каждый возьмёт, обещают, и/или обсуждают третьих агентов (потенциальный
   gossip). Каждая реплика несёт флаг `ready`. Правило остановки — параметр
   `talk_stop_rule`, дефолт `both_ready_latch` (сказал «готов» — выходишь из
   разговора, дозревает второй; альтернативы `both_ready_revocable` / `either` /
   `fixed_k`). Сверху всегда жёсткий потолок `max_talk_turns`; минимум — по одной
   реплике с каждого; кто начинает — определяется сидом.
3. **Решение.** Каждый агент **приватно и одновременно** выбирает число 0–9
   (структурированный вывод `{number, rationale}`). Обоснование приватно —
   партнёру не раскрывается, но пишется в артефакт для анализа.
4. **Вскрытие и платёж.** Числа раскрываются, считается исход и платежи (ниже),
   обновляются кумулятивные очки.
5. **Рефлексия (опц., `REFLECT`).** Если `reflect_after` включает пару — каждый
   агент приватно «думает» над итогом (партнёру не видно); пишется в артефакт и
   в его память. Тут вероятно формируются намерения (отомстить, предупредить).
6. **Обновление памяти.** Каждому агенту в память добавляется: раунд, ID
   партнёра, полный cheap-talk, своё число + своё обоснование, число партнёра,
   исход, платёж, своя рефлексия. (Чужое обоснование/рефлексия НЕ раскрываются.)
   Раз в `notes_every_n` раундов — фаза `NOTES` (см. §6).

**Исход по паре** (X — число агента, Y — партнёра; дефолт R=3, T=5, P=1, S=0):

| Условие | Исход | Платёж |
|---|---|---|
| `X == Y` | оба честны (CC) | (R, R) = (3, 3) |
| `X == (Y+1) mod 10` | X обманул (DC) | X: T=5, Y: S=0 |
| `Y == (X+1) mod 10` | Y обманул (CD) | Y: T=5, X: S=0 |
| иначе (70 из 100 пар) | взаимная защита (DD) | (P, P) = (1, 1) |

Свойства: T>R>P>S и 2R>T+S соблюдены — строгое PD. Это направленный цикл
(`1>0, 2>1, …, 9>8, 0>9`): каждое число эксплуатирует ровно одно (на 1 ниже по
кругу). Off-by-one не бывает взаимным → в паре максимум один получает T, исход
всегда однозначен.

**Тонкость стратегии (фича, не баг).** Чтобы получить T, надо угадать число
партнёра и взять ровно +1. Если оба наивно пытаются обмануть, оба берут `k+1` →
снова равенство → (R, R): взаимная попытка предательства схлопывается в
кооперацию. Это даёт надстройку «уровневого» мышления (beauty contest mod 10).
Неточный обман (не ровно +1) наказывается до P, а не платит T.

**Как тут возникает репутация.** Агент A знает о поведении C только из (а) личного
опыта (играл с C ранее) или (б) услышанного про C. В cheap talk A может передать B
инфу про C («C обещал взять 5, а взял 6, чтобы меня кинуть») — правдивую или
ложную. B может позже это использовать. Эмерджентность репутации = появляется ли
такой обмен и влияет ли он на решения. (Измерение — §12, отложено.)

## 6. Память агента (два яруса)

Рендер памяти в промпт = `[notes] + [последние W раундов сырьём]`.

- **Сырой ярус — контекстное окно.** Последние W раундов дословно (свои cheap-talk,
  числа, исходы, рефлексии). `context_window` = W в раундах; ∞ = полная история
  (дефолт). Дополнительно опциональный токен-кап как предохранитель.
- **Ярус notes.** Раз в `notes_every_n` раундов агент сам обновляет свободные
  заметки (фаза `NOTES`); они всегда в контексте и переживают выпадение старого
  из окна. `notes_every_n` = 0 → выключено. Обновление notes происходит, пока
  выпадающие записи ещё в окне, — чтобы агент мог сохранить уходящее.
- Эти два параметра **субсумируют** прежние варианты: raw = (W=∞, notes off);
  truncated = (W=k, notes off); самосводка = (W малое, notes on). Оба — свип-оси.

Важно:
- **Исследовательски:** узкое окно ослабляет прямую взаимность и делает notes/слухи
  несущими → окно изолирует эффект репутации.
- **Нейтральность формулировок:** промпт notes держим нейтральным («запиши, что
  хочешь помнить»), НЕ «запиши, кто тебя кинул» — иначе наполовину построим
  институт репутации за агентов и сместим результат. То же для `REFLECT`.
- **Возобновление:** сырой ярус реконструируется из транскриптов; notes и рефлексии
  — *сгенерированные* артефакты, поэтому чекпойнтятся и восстанавливаются из
  чекпойнта (регенерировать их нельзя без повторного вызова LLM).

## 7. Провайдеры LLM

- Протокол `LLMProvider.complete(*, system, messages, temperature, max_tokens) ->
  Completion(text, usage, raw)`. Async.
- **Единый `OpenAICompatibleProvider`** с настраиваемыми `base_url` + `api_key` +
  `model`. Все рассматриваемые бесплатные сервисы (локальная Ollama, Cerebras,
  Gemini, Groq, OpenRouter, Mistral) и OpenAI/Anthropic отдают OpenAI-совместимый
  endpoint → один клиент покрывает всё; «провайдер» = строка `base_url`+`model` в
  конфиге агента. Это же даёт многомодельность (разным агентам — разные endpoint/model).
- **Дефолт MVP:** `base_url` → локальная Ollama (`qwen3:8b`), бесплатно и без
  лимитов. Для «настоящих» прогонов — модель посильнее сменой конфига
  (Cerebras / Gemini / Claude); мелкие локальные модели могут быть слабоваты для
  тонкой социальной динамики.
- Ключи из env (`*_API_KEY`); для локальной Ollama — ключ-заглушка.
- Ретраи с экспоненциальным backoff на 429/5xx; учёт токенов в каждом `Completion`.
- **Пост-MVP:** нативный `AnthropicProvider` с prompt caching, если Claude будет
  использоваться объёмно.

## 8. Оркестрация: эпохи → раунды + эволюционный отбор

Иерархия: **эпизод = `num_epochs` эпох × `epoch_length` раундов** (эпохи выключены
при `num_epochs=1` → плоские R раундов, текущее поведение).

- **Раунд** = две фазы: (1) **матчинг** — `plan = await matchmaker.plan_round(...)`
  (может быть интерактивным, зовёт агентов через `actor`); (2) **игра** — прогон
  пар из `plan.pairings` через `game.play_pairing`. Пар в раунде — от 1 (var2) до N/2.
- **Независимые пары — параллельно** (`asyncio.gather` + семафор `max_concurrency`);
  каждая пара — последовательная цепочка. Интерактивный матчинг обычно последователен.
- `idle`-агенты получают `idle_payoff`. **Раунды последовательны** (память зависит
  от прошлых). `matchmaker.setup(...)` вызывается **в начале каждой эпохи** (ростер
  меняется отбором → расписание `scheduled` генерируется пер-эпоха).

**Эволюционный отбор (между эпохами, `num_epochs−1` раз):**
- **Фитнес** — кумулятивные очки агента за завершившуюся эпоху; **очки обнуляются
  в начале каждой эпохи**. Честно по построению: потомок добавляется на границе
  эпох и не играет в эпохе E; в E+1 все (выжившие и потомки) стартуют с 0 и
  отыгрывают полную эпоху → сравнение на равных. (Альтернатива — `fitness_metric`:
  пожизненный кумулятив без обнуления.)
- **Отбор:** K слабейших удаляются; K сильнейших дают по одному потомку.
  `K = round(selection_frac·N)`, `K ≤ N/2` (чтобы топ и низ не пересекались);
  `parent_policy`: top-K-каждый / пропорц. фитнесу / лучший×K.
- **Потомок = копия `AgentSetup` родителя** (provider/model/persona/temp), но
  **новый анонимный ID и пустая память/notes/очки** — генотип, не полная копия.
  Для других он чужак без репутации. Родословная пишется вне игры (для анализа),
  агентам не видна (`reveal_lineage`, дефолт off).
- Выжившие сохраняют память/notes/очки между эпохами; убитые удаляются (их
  транскрипты остаются в истории, упоминания их ID в чужой памяти — исторические).

- **Чекпоинт после каждого раунда И после отбора**: `rounds.jsonl`, `epochs.jsonl`
  (ростер, фитнес, убитые/рождённые, родословная), транскрипты → возобновляемо.
- **Пользовательский останов**: graceful (флаг/сигнал) — доигрывает текущий раунд,
  флашит артефакты, помечает статус.

## 9. Свип по параметрам и конфиги

- Конфиг эксперимента в YAML: базовые параметры + списки значений для свипа.
- Оси свипа (из требований): модели агентов (в т.ч. разные у двоих в паре),
  вариант промпта/персоны, и «лёгкая вариация правил» — матрица платежей,
  число раундов R, размер популяции N, `max_talk_turns`, `talk_stop_rule`,
  схема знания ростера,
  память (`context_window`, `notes_every_n`), пост-игровая рефлексия
  (`reflect_after`).
- **Состав популяции — отдельная свип-ось (`PopulationGenerator`, `population.kind`):**
  как собран стартовый ростер. Сюда сходятся оси «модели агентов», «вариант
  персоны», «размер N» из пункта выше.
  - `roster` — явный список персон/провайдеров, циклически до N (подпараметры
    `n_agents`, `agents[]`).
  - `homogeneous` — N идентичных слотов из одной персоны+модели (`n_agents`).
  - `mixed` — тянуть персону/модель из пулов по правилу (`even` / `random`); сид фиксирует выбор.
- **Схема матчинга — отдельная (ключевая) свип-ось:**
  - `random` — случайная разбивка каждый раунд.
  - `scheduled` (var1) — предгенерированное согласованное расписание пар;
    подпараметры `disclose` (none / next_k / full — «агент знает свою
    последовательность» = full), `schedule_gen` (random / round-robin).
  - `choice` (var2) — выбор партнёра агентом; подпараметры `chooser_policy`
    (rotation / random / all_sequential), `candidate_set` (все / подвыборка k),
    `allow_refusal` (вкл/выкл), `on_refusal` (повтор m / idle / следующий),
    `idle_payoff` (≤ P).
- **Эволюционные эпохи — отдельная свип-ось:** `num_epochs` (1 = выкл),
  `epoch_length`, `selection_frac` (≤ 0.5), `fitness_metric` (epoch-cumulative с
  обнулением [дефолт] / lifetime-cumulative), `parent_policy` (top-K / proportional
  / best×K), `reveal_lineage` (off/on).
- Разворачивание: декартово произведение свип-осей × `repeats` повторов, каждый
  повтор со своим сидом.
- `run_id` = целочисленный автоинкремент (1, 2, 3 …); идентичность прогона — это его номер.
  «Дизайн» (для группировки повторов/продолжений в семью) хешируется отдельно в
  `config_hash` = хеш конфига без `judge` и без `rounds`.

## 10. Хранилище артефактов — SQLite

Одна БД SQLite (stdlib, без сервера, файл) — держит много прогонов (под свип).
Транскрипты — JSON-колонкой (verbatim). SQL удобен для возобновления, метрик,
свипа и анализа репутации. (DuckDB — опц. пост-MVP под тяжёлую аналитику.)

Таблицы (ключи упрощённо):
- `runs` — `run_id` (PK), `experiment_id`, `config` (JSON), `seed`, `status`,
  модели, хеши шаблонов промптов, хеш кода, `started_at`/`finished_at`.
- `agents` — (`run_id`, `agent_id`), `setup` (JSON: persona, base_url, model),
  очки (финальные / по эпохам), родословная `parent_id` + эпоха рождения (пост-MVP).
- `rounds` — (`run_id`, `round_idx`), `epoch_idx`, `plan` (JSON: пары + idle +
  события матчинга).
- `pairings` — (`run_id`, `round_idx`, `pair_idx`), `a_id`, `b_id`, `a_number`,
  `b_number`, `a_rationale`, `b_rationale`, `outcome`, `a_payoff`, `b_payoff`,
  `transcript` (JSON: реплики+ready), приватные рефлексии (JSON, пост-MVP),
  `usage` (JSON).
- `epochs` — (`run_id`, `epoch_idx`): ростер, фитнес, убитые/рождённые,
  родословная (пост-MVP).
- метрики — отдельной таблицей или считаются SQL-запросами на лету.

- **Запись без гонок:** пары раунда играются параллельно, но пишутся **одной
  транзакцией после `gather()`** (раунды последовательны) → single-writer SQLite
  не конфликтует. WAL-режим для чтения во время анализа.
- **Воспроизводимость:** в `runs` — полный конфиг, сид, версии моделей, хеши
  шаблонов промптов, хеш кода.

## 11. Авто-метрики на прогон (считаем сразу, дёшево, без LLM-судьи)

- Доля кооперации (равные числа) по раундам и итоговая; доля удачных обманов
  (off-by-one) и доля раскоординации («иначе»).
- Распределение платежей: итоговые очки, разброс/Gini, ранжирование агентов.
- Длины сообщений (символы/токены), суммарный расход токенов и стоимость.
- **Структурный прокси gossip**: доля реплик, упоминающих ID агента, не входящего
  в текущую пару (по списку известных ID). Дёшево, без LLM, и прямо релевантно
  вопросу репутации. (Полноценный LLM-судья — §12.)

## 12. Анализ репутации — ОТЛОЖЕН

Отдельный модуль `analysis/reputation.py` (сейчас только интерфейс-заглушка):
- LLM-судья тегирует реплики: упоминание третьего, наличие репутационной инфы,
  её валентность, цель, правдивость (сверка с ground truth действий).
- Поведенческая корреляция: меняет ли «услышанный слух» про C решение агента
  против C при следующей встрече.
Таблица `pairings` (§10) уже содержит всё нужное (публичный текст отдельно от
приватных обоснований, действия — ground truth) и удобна для SQL-запросов.
Реализуем после согласования метрики.

## 13. Параллелизм, rate-limit, возобновляемость

- Параллельны: пары внутри раунда; эпизоды внутри свипа (оба под общим семафором).
- Rate-limit: `max_concurrency` + backoff-ретраи в провайдере.
- Возобновляемость: пропуск эпизодов со `status='done'` в таблице `runs`; внутри
  эпизода — продолжение с `MAX(round_idx)+1` из `rounds` (память реконструируется
  реплеем `pairings` до этой точки).

## 15. Порядок реализации (этапы) + runbook

1. Скелет: конфиг-датаклассы, провайдер (base + `OpenAICompatibleProvider`),
   `Agent.act(phase)`, двухъярусная память (окно W + хранение notes), хранилище.
2. Игра `reputation_pd`: фазы `TALK` (с `ready`) + `DECIDE` (число) + платежи;
   `PairingRecord`; опц. фаза `REFLECT`.
3. Матчинг + оркестратор: `Matchmaker` (сначала `random`, затем `scheduled`,
   `choice`); цикл раундов (матчинг → игра), параллельные независимые пары,
   периодические `NOTES`, прокладка памяти, чекпоинты.
4. Эпохи + отбор: `population` + `SelectionPolicy` (фитнес, K, размножение от
   `AgentSetup` со свежими ID, родословная, `epochs.jsonl`); эпоха-обёртка над
   циклом раундов.
5. Свип + CLI: разворачивание сетки, сиды/повторы, возобновляемость, статус,
   `estimate` (оценка числа вызовов и токенов до запуска).
6. Авто-метрики (включая структурный прокси gossip).
7. Smoke-тест: крошечный прогон (N=4, R=3, `max_talk_turns`=2, матчинг `random`,
   `num_epochs`=1) на дешёвых моделях; затем мини-прогон с `num_epochs`=2 и
   `scheduled`/`choice`; проверить артефакты и метрики.
8. (Отложено) модуль анализа репутации после согласования метрики.

Runbook: задать `config/example.yaml` → `cli estimate` (прикинуть объём/стоимость)
→ `cli run` → следить за прогрессом → `cli status` → анализ артефактов.

## 16. Открытые поддекизии (предлагаемые дефолты — уточнить на ревью)

- `max_talk_turns` (жёсткий потолок) дефолт = 6; `talk_stop_rule` дефолт
  `both_ready_latch`; минимум по одной реплике с каждого; первый говорящий — по сиду.
- Публичные сообщения и приватные обоснования храним раздельно; приватное
  обоснование агента возвращается ему в его же память, партнёру — никогда.
- Знание ростера: дефолт — агент знает N и свой ID, чужие ID узнаёт по контакту
  или из слухов; `reveal_roster` (выдать весь список ID сразу) — параметр.
- Память: `context_window` в раундах (дефолт ∞) + опциональный токен-кап;
  `notes_every_n` (дефолт 0 = выкл). Промпты `NOTES`/`REFLECT` — нейтральные.
- Рефлексия: `reflect_after` дефолт = `pairing` (off / pairing / round / episode).
- Финальное решение принимается независимо обоими (никто не видит выбор другого
  до вскрытия) — это и есть «одновременность».
- Нечётный N или непарный по итогам матчинга → idle, платёж `idle_payoff`
  (дефолт = P).
- Матчинг `choice` (дефолты): `chooser_policy`=rotation, `candidate_set`=все,
  `allow_refusal` — обе ветки как вариант, `on_refusal`=один повтор затем idle.
- Матчинг `scheduled` (дефолты): `disclose`=full, `schedule_gen`=random без
  немедленного повтора той же пары.
- Эпохи (дефолты): `num_epochs`=1 (отбор выкл), `selection_frac`≈0.2 (K≤N/2),
  `fitness_metric`=epoch-cumulative с **обнулением очков в начале эпохи**,
  `parent_policy`=top-K, `reveal_lineage`=off.
- **Фиксировано (не параметр):** агенту не сообщается про эпохи/отбор/выбраковку —
  он просто максимизирует платёж, эволюция действует извне.
- Тех. `max_tokens` на вызов как предохранитель, хотя канал «без ограничений».

## 17. Риски и стоимость

- Всего раундов R = `num_epochs` × `epoch_length`. Число LLM-вызовов на эпизод ≲
  (N/2 пар) × R × (max_talk_turns×2 + 2 решения + 2 рефлексии) +
  (N × R / notes_every_n) на notes — верхняя оценка; реально меньше из-за раннего
  `ready`. Растёт быстро при увеличении N, R, лимита реплик; `reflect_after`
  удваивает «хвост» на пару, `notes` добавляет периодические. Сам отбор —
  без LLM (фитнес из записанных платежей); интерактивный матчинг `choice`
  добавляет вызовы на выбор/согласие.
- Контроль: команда `estimate` печатает прогноз вызовов/токенов/стоимости ДО
  запуска; старт с дешёвых моделей и малых N/R/K; prompt caching; `max_concurrency`.
- LLM недетерминирован (даже при temperature=0) — фиксируем сид матчинга/персон,
  пишем всё дословно; точную воспроизводимость по тексту не гарантируем.
- Сырая память распухает на длинных прогонах — смягчается параметром
  представления памяти (§6) и `truncated_k`.
- Скаффолдинг памяти/ростера способен сместить результат об эмерджентности —
  поэтому держим эти вещи параметрами, а дефолты — минимально навязчивыми.

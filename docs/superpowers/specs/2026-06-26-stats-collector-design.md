# Дизайн: judge-backfill + сборщик статистики + визуализация

Дата: 2026-06-26

## Задача

В реплей-БД (`experiment.db`) лежат полные прогоны (на момент написания — 100 прогонов,
77 завершённых), но вердиктов судьи нет (`judge_verdicts` пуста): судья сейчас запускается
только вживую в конце свежего эпизода — не при resume и не для уже сохранённых прогонов.

Нужно:
1. Оценить выбранные сохранённые прогоны LLM-судьёй и записать вердикты.
2. Собрать статистику по оценённым прогонам: доля «возникновения института репутации»
   (emergence rate) по дизайнам эксперимента, с доверительными интервалами.
3. Отдельным скриптом визуализировать результат со «статистическими ошибками» (CI).

Выбор прогонов должен быть **опциональным/фильтруемым** — какие прогоны включать, какие нет.

## Архитектура

Три тонких скрипта верхнего уровня (в стиле существующих `experiment.py` / `replay.py`),
опирающихся на маленькие **чистые, юнит-тестируемые** модули в `src/`. Конвейер линейный:

```
experiment.db ──▶ judge_runs.py ──▶ таблица judge_verdicts
                                          │
                  collect_stats.py ◀──────┘
                       │   └──▶ stats.json + stats.csv (артефакты)
                       ▼
                  plot_stats.py ──▶ stats.png
```

Разделение ролей (решение пользователя — три отдельных инструмента):

- **`judge_runs.py`** — backfill: выбрать сохранённые прогоны, восстановить их публичный
  cheap-talk из БД, вызвать существующий `judge_episode`, записать вердикты
  (уже оценённые пропускаются, если не задан `--force`).
- **`collect_stats.py`** — агрегация: выбрать прогоны, у которых есть вердикт, сгруппировать
  по дизайну (`config_hash`), посчитать emergence rate + 95% интервал Вилсона, напечатать
  таблицу, записать `stats.json` и `stats.csv`.
- **`plot_stats.py`** — визуализация: прочитать `stats.json`, построить столбчатую диаграмму
  (matplotlib) с асимметричными усами интервала Вилсона, сохранить PNG.

Принцип владения выводом сохраняется: вся «настоящая» логика (выборка, агрегация, интервал,
реконструкция записей) живёт в `src/` и ничего не печатает; печать/CLI/файлы — в скриптах
верхнего уровня.

## Компоненты

### Новый код в `src/` (чистая логика, TDD)

- **`src/stats/wilson.py`** — `wilson_interval(k, n, z=1.96) -> tuple[float, float]`.
  Замкнутая формула интервала Вилсона для доли, без scipy. Единственная «настоящая»
  математика, покрывается юнит-тестами (k=0, k=n, малое n, середина).

  Формула (доля `p = k/n`, по умолчанию z=1.96 для 95%):
  ```
  denom  = 1 + z^2/n
  center = (p + z^2/(2n)) / denom
  half   = (z/denom) * sqrt( p(1-p)/n + z^2/(4 n^2) )
  lo, hi = clamp(center - half, 0, 1), clamp(center + half, 0, 1)
  ```
  `n == 0` — интервал не определён (вызывающий такие группы не передаёт; см. ниже).

- **`src/stats/selection.py`** — общий `RunFilter` (frozen dataclass): include/exclude по
  `config_hash` и по `name`, плюс `finished_only=True`. Используется **и** backfill, **и**
  сборщиком, чтобы «какие прогоны» означало одно и то же везде. Функция
  `selected_run_ids(conn, flt) -> list[int]` возвращает номера прогонов, прошедших фильтр
  (порядок — по `created_at`).

  Семантика фильтра: include-списки (если непусты) — белый список; exclude-списки —
  чёрный список поверх; `finished_only` отбрасывает прогоны без `finished_at`.

- **`src/stats/aggregate.py`** — frozen dataclasses и агрегация:
  - `RunRow(run_id, config_hash, name, emerged)` — один оценённый прогон.
  - `DesignStat(config_hash, name, n, n_emerged, rate, ci_lo, ci_hi, run_ids)`.
  - `load_judged_runs(conn, run_ids) -> list[RunRow]` — JOIN `runs` × `judge_verdicts` по
    выбранным номерам (прогоны без вердикта молча выпадают).
  - `aggregate_by_design(rows) -> list[DesignStat]` — группировка по `config_hash`,
    `k = Σ emerged`, `n = len(group)`, `rate = k/n`, интервал из `wilson_interval`.
    `name` группы — `name` первого прогона в ней (ярлык, для подписи на графике).

  Чистый SQL-чтение + арифметика; тесты — против in-memory SQLite с реальной схемой.

- **`src/storage/records.py`** — `reconstruct_records(conn, run_id) -> list[ReplayRecord]`:
  восстановить минимальные записи **завершённых** пар (round, **истинный** pair_idx, a_id,
  b_id, transcript из `messages`), которые нужны судье. `ReplayRecord` — лёгкий frozen
  dataclass с полями `round`, `pair`, `a_id`, `b_id`, `transcript`, `finished=True`.

### Изменения в существующем `src/`

- **`src/judge/transcript.py`** — сделать `render_transcript` и `valid_refs` **pair-aware**:
  если у записи есть явный `.pair` (не `None`) — использовать его, иначе нумеровать позицией
  (как сейчас). Обратносовместимо: живой путь судьи (`PairingRecord` без `.pair`) работает
  как прежде (нумерация), backfill передаёт `ReplayRecord` с истинным `pair_idx` →
  доказательства-ссылки `r.p.t` совпадают с поиском по `messages` в `replay.py`.
  Это сохраняет **единый** путь судьи (`judge_episode` не меняется).

  > Почему это решение: при наличии сорванной пары в раунде живой судья перенумеровывает
  > `p` (видит только завершённые пары), и evidence-ссылка сместилась бы относительно
  > `pair_idx` в БД. Backfill хранит истинный `pair_idx`, чтобы подсветка в `replay.py` всегда
  > указывала на правильное сообщение.

- **`src/storage/store.py`** — два небольших добавления для backfill (одна `Storage`
  оценивает много прогонов):
  - `has_verdict(run_id) -> bool`.
  - `save_verdict(verdict, *, model, run_id=None)` — необязательный `run_id` адресует
    конкретный прогон (по умолчанию — текущий `_run_id`, как сейчас).

### Зависимости

- `uv add matplotlib` (numpy подтянется транзитивно). Интервал Вилсона считается вручную —
  scipy не нужен.

## CLI

Общие флаги фильтра — у backfill и сборщика (один и тот же `RunFilter`):

```
uv run python judge_runs.py    [--design HASH ...] [--exclude-design HASH ...] \
                               [--name LABEL ...]  [--exclude-name LABEL ...] [--force]
uv run python collect_stats.py [те же фильтры] [--out stats.json] [--csv stats.csv]
uv run python plot_stats.py    [--in stats.json] [--out stats.png] [--title TEXT]
```

Без фильтров — все подходящие прогоны (для backfill: все завершённые без вердикта;
для сборщика: все оценённые).

Конфиг судьи для backfill следует шаблону `experiment.py`: редактируемая константа уровня
модуля `JUDGE = JudgeCfg(...)`. Для каждого прогона судья берётся так: если в сохранённом
конфиге прогона есть блок `judge` — используется он; иначе — `JUDGE` по умолчанию из скрипта.
Модель судьи пишется в `judge_verdicts.model`.

## Поток данных и статистика

- Один прогон даёт один бинарный исход `emerged ∈ {0,1}` из `judge_verdicts`.
- Группировка по `config_hash`; в группе `k = Σ emerged`, `n = #оценённых прогонов`,
  `rate = k/n`.
- **95% интервал Вилсона** для доли — корректен у краёв 0/1 и при малом `n` (на дизайн —
  порядка десятков прогонов). Усы асимметричны: нижний `rate − ci_lo`, верхний `ci_hi − rate`.

### Артефакт `stats.json`

```json
{
  "filters": { "include_designs": [], "exclude_designs": [],
               "include_names": [], "exclude_names": [], "finished_only": true },
  "designs": [
    { "config_hash": "ab12…", "name": "baseline", "n": 24, "n_emerged": 17,
      "rate": 0.708, "ci_lo": 0.508, "ci_hi": 0.852,
      "run_ids": [3, 7, 9, ...] }
  ]
}
```

### Артефакт `stats.csv`

Плоская таблица, по строке на дизайн (для импорта в таблицы/R):
`config_hash,name,n,n_emerged,rate,ci_lo,ci_hi`.

## Обработка ошибок

- **backfill**: сбой судьи на одном прогоне — русское предупреждение и продолжаем (как
  `_judge_and_store` в runner); незавершённые прогоны пропускаются с уведомлением.
- **сборщик**: дизайн с `n == 0` оценённых прогонов в артефакт не попадает (предупреждение);
  пустая выборка — внятное сообщение и ненулевой код выхода.
- **визуализация**: отсутствующий/пустой `stats.json` — внятная русская ошибка.

## Тестирование (TDD, по конвенциям проекта)

- `tests/stats/test_wilson.py` — проверки интервала по известным значениям (k=0, k=n,
  малое n, середина); монотонность сужения с ростом n.
- `tests/stats/test_selection.py` — in-memory БД (реальная схема), проверка include/exclude
  по design/name и `finished_only`.
- `tests/stats/test_aggregate.py` — seed БД прогонами+вердиктами вручную, проверка
  группировки/`rate`/CI и что прогоны без вердикта выпадают.
- `tests/storage/test_records.py` — реконструкция отдаёт ровно завершённые пары с истинным
  `pair_idx` и сообщениями в порядке `turn_idx`.
- `tests/judge/test_transcript.py` — расширить: pair-aware рендер использует `.pair`, а без
  него нумерует позицией (обратная совместимость живого пути).
- backfill: юнит-тест с `ScriptedProvider` — сохранённый прогон получает строку вердикта;
  поведение skip-if-exists и `--force`.
- `plot_stats.py` — тонкий рендер; smoke-тест (пишет PNG во временный путь, без проверки
  пикселей; matplotlib backend `Agg`).

## Вне области (YAGNI)

- Никаких дополнительных метрик помимо бинарного вердикта (CC-доля, средние счёта,
  распределение исходов) — заголовочная статистика именно про вердикт судьи.
- Никакой группировки по `name` отдельно от `config_hash` — `name` лишь подпись группы.
- Никакой интерактивной/веб-визуализации — один PNG.

Дата: 2026-06-04. Четвёртый слой разработки (снизу вверх), стоит на готовых слоях
провайдера, Агента и Игры. Опирается на `agent-games-mvp-arch.md` §3
(`matchmaking/base.py`, `random_mm.py`), §4 (поток раунда), §8 (швы).
**При изменениях контракта** (`RoundPlan` / `Matchmaker` / `make_matchmaker`) —
синхронно править arch. Идём **срезами** (§6).

**Язык промптов: английский** (зафиксировано 2026-06-04). На этот слой не влияет —
матчинг чисто алгоритмический, без вызовов LLM. Запись для контекста: реплики
появятся в интерактивных матчингах (`choice`/`consent`, пост-MVP) и в оркестраторе.

## 0. Зачем Матчинг четвёртым

Слой 4 в `-explained` — «кто с кем играет в раунде». Игра умеет провести **одну**
пару (`play_pairing`). Матчинг превращает популяцию (список живых ID) в набор
**непересекающихся** пар на раунд. Это последний слой перед оркестратором:
оркестратор затем лишь склеит «популяция → матчинг → параллельные партии → хранилище».

Ключевое свойство, ради которого матчинг — отдельный слой: **разбиение на
непересекающиеся пары есть ровно то, что делает параллельную игру безопасной без
блокировок** — две пары никогда не трогают одного агента → нет гонок по памяти/`score`
(mvp-arch §4). Матчинг обязан этот инвариант гарантировать; всё остальное в нём
тривиально.

## 1. Анализ: что нетривиально

Само случайное паросочетание примитивно (перемешать, разбить попарно). Нетривиальны
**контракт и его инварианты** — то, что должно остаться стабильным, когда добавим
интерактивные матчинги.

1. **Инвариант разбиения (главный).** Для входного списка `ids`: каждый ID попадает
   **ровно в одну** пару **или** в `idle`; никогда дважды; никогда и там и там.
   Формально: `flatten(pairings) ∪ idle == set(ids)`, три множества попарно
   дизъюнктны, в каждой паре два **разных** ID. Это THE инвариант — на нём держится
   безгоночный параллелизм оркестратора. Тестируем исчерпывающе.

2. **Нечётная популяция → один `idle`.** При нечётном N один агент сидит вне игры и
   получает `idle_payoff`. Кто именно — последний после перемешивания (т.е.
   случайный по `rng` → в среднем честно ротируется между раундами). **`idle_payoff`
   начисляет оркестратор, не матчинг** (mvp-arch §3:
   `for c in plan.idle: pop.get(c).score += cfg.idle_payoff`). Матчинг лишь
   **сообщает** `idle`. Гарантированная равная ротация «каждый сидит ровно k раз» —
   шов пост-MVP (§7).

3. **Воспроизводимость через `rng` (РЕШЕНО, §8).** Матчер держит **собственный
   производный** `rng` (`Random(f"{seed}:matchmaker")`), заведённый в `setup` — **не**
   общий с играми. Один seed → одна и та же последовательность разбиений по раундам,
   **независимо** от того, что творится в партиях и как их тасует `gather`. Это
   **единственная** воспроизводимость, которую мы хотим: внутри партии её не
   преследуем (LLM недетерминирован при temperature>0), `Game.play_pairing` `rng`
   вообще не принимает. Так матчер становится единственным источником **структурной**
   случайности (и кто с кем, и — через ориентацию пары, §1.4 — кто открывает).

4. **Ориентация пары (кто `a`, кто `b`) — ЗНАЧИМА.** Пара — кортеж `(id1, id2)`;
   порядок становится `(a, b)` в `play_pairing` и определяет две вещи: (1) **кто
   открывает cheap-talk** — `a` (Игра монетку не кидает, `rng` ей не нужен — §1.3,
   game-plan §14.6); (2) с чьей точки зрения пишется `outcome` в `PairingRecord`.
   Поскольку ориентацию задаёт перемешивание матчера на его **воспроизводимом** `rng`,
   первый говорящий тоже воспроизводим по seed. Чтобы сделать `b` открывающим —
   матчер возвращает `(b, a)`.

5. **`setup` vs `agent_ids` в `plan_round` — зачем оба.** Популяция пост-MVP меняется
   между раундами (эволюция: `add`/`remove`). `setup` — разовая инициализация
   (сохранить `rng`/`cfg`; для будущего `scheduled` — предпосчитать расписание по
   стартовым ID). `plan_round` получает **текущие** живые ID каждого раунда (могут
   отличаться). RandomMatchmaker: `setup` лишь прячет `rng`; `plan_round` перемешивает
   то, что дали сейчас. Документируем как шов под `scheduled`/`choice`.

6. **`actor`-колбэк в `plan_round`.** Параметр `actor` — колбэк вызова агента для
   интерактивных матчингов (`choice`: кто кого выбирает; `consent`: согласие).
   RandomMatchmaker его **игнорирует**. Он в сигнатуре, чтобы `Protocol` не менялся,
   когда добавим интерактивные матчинги (mvp-arch §8). Шов.

7. **`events` в `RoundPlan`.** `events: list[dict]` — шов для богатых матчингов (кто
   кому предложил, кто согласился). RandomMatchmaker → `events=[]`. Оркестратор пишет
   `plan` (включая `events`) JSON-колонкой в `rounds` (mvp-arch §6). В MVP пусто.

8. **`make_matchmaker(kind)` — фабрика.** Выбор реализации по строке `cfg.matchmaker`
   (`"random"`), зеркало `make_population`. RandomMatchmaker параметров не требует
   (`rng` приходит в `setup`). Будущие (`"scheduled"`, `"choice"`) — тот же
   `Protocol`; их параметры потекут через `setup(cfg)`, фабрика остаётся тупой
   (берёт только `kind`). Неизвестный `kind` → `ValueError` с внятным сообщением.

9. **Краевые N.** `N=0` → пустой план. `N=1` → `idle=[единственный]`, `pairings=[]`.
   `N=2` → одна пара, `idle=[]`. Чётный N → `idle=[]`. Нечётный N≥3 → `(N-1)/2` пар +
   1 `idle`. Все — в исчерпывающих тестах.

## 2. Границы слоя (scope)

**В MVP:**
- `RoundPlan` (dataclass), `Matchmaker` (Protocol), `make_matchmaker(kind)` в `matchmaking/base.py`.
- `RandomMatchmaker` (`setup` + `plan_round`) в `matchmaking/random_mm.py`.
- Тесты: исчерпывающий инвариант разбиения (N=0..5); детерминизм по seed; `idle` ⇔
  нечётность; дизъюнктность; различность ID в паре; контракт-тест швов
  (`actor`/`events`/`setup`); интеграционный микротест matching→game (одна пара через
  `ScriptedProvider`) **без** оркестратора.

**НЕ в MVP (швы, §7):**
- Матчинги `scheduled` / `choice` / `consent` (интерактивные).
- Гарантированная равная ротация `idle`.
- `Population` (мутабельный ростер) и `make_population` — это инфраструктура слоя
  Оркестратора (слой 5); здесь матчинг работает на голом `list[str]`.
- `EpisodeCfg`, цикл раундов, хранилище, метрики, CLI — слой Оркестратора.

## 4. Интерфейсы (сигнатуры) — целевое, из arch §3

```python
# matchmaking/base.py
@dataclass
class RoundPlan:
    pairings: list[tuple[str, str]]   # непересекающиеся пары (a_id, b_id)
    idle: list[str]                   # кто сидит вне игры (нечётный N) — 0 или 1 в MVP
    events: list[dict]                # шов под интерактивные матчинги; random → []

class Matchmaker(Protocol):
    def setup(self, agent_ids: list[str], rng, cfg) -> None: ...
    async def plan_round(self, agent_ids: list[str], round: int, actor) -> RoundPlan: ...
    #   actor — колбэк вызова агента (интерактивные матчинги); random не использует

def make_matchmaker(kind: str) -> Matchmaker:    # "random" -> RandomMatchmaker(); иначе ValueError
    ...
```

```python
# matchmaking/random_mm.py
class RandomMatchmaker:
    def setup(self, agent_ids, rng, cfg) -> None     # сохранить rng (и cfg при нужде)
    async def plan_round(self, agent_ids, round, actor) -> RoundPlan
    #   перемешать копию agent_ids → пары подряд; нечётный → последний в idle; events=[]
```

## 5. Алгоритм `RandomMatchmaker`

```python
def setup(self, agent_ids, rng, cfg):
    self._rng = rng                       # (опц.) self._cfg = cfg для будущих параметров

async def plan_round(self, agent_ids, round, actor):
    ids = list(agent_ids)                 # копия: вход не мутируем
    self._rng.shuffle(ids)
    idle = [ids.pop()] if len(ids) % 2 else []     # последний после перемешивания
    pairings = [(ids[i], ids[i + 1]) for i in range(0, len(ids), 2)]
    return RoundPlan(pairings=pairings, idle=idle, events=[])
```

Замечания: `shuffle` — на **копии** (не мутируем вход). `plan_round` **async** по
`Protocol`, но тело синхронно (нет `await`) — это нормально: async ради единого
контракта с интерактивными матчингами. `round`/`actor` RandomMatchmaker не использует
(нужны сигнатуре, логам, будущему `scheduled`).

## 6. Срезы (порядок реализации)

**Срез 1 — типы + фабрика-скелет.**
- `RoundPlan`; `Matchmaker` (Protocol); `make_matchmaker("random") → RandomMatchmaker()`,
  неизвестный `kind` → `ValueError`. `RandomMatchmaker.setup` (хранит `rng`).
- Тест: фабрика возвращает нужный тип; `ValueError` на мусорный `kind`.
- **DoD:** импортится, фабрика зелёная.

**Срез 2 — `plan_round` + инвариант разбиения.**
- Реализовать shuffle / pair / idle.
- Тесты исчерпывающе, N=0..5: partition-инвариант (объединение == set(ids),
  дизъюнктность, различность ID в паре), `idle` ⇔ нечётность, размер `pairings`.
- **DoD:** инвариант зелёный на всех N; вход не мутируется.

**Срез 3 — детерминизм + швы + интеграция с Игрой.**
- Детерминизм: один seed → идентичные планы по раундам; разные seed → (как правило)
  разные.
- Контракт швов: `plan_round` принимает `actor` (игнор) и работает; `events == []`;
  `setup` принимает `cfg` без падения.
- **Интеграционный микротест** matching→game: 2–3 `ScriptedProvider`-агента (ID —
  как у будущего ростера, напр. `A1..A3`) → `plan_round` → для каждой пары
  `game.play_pairing(...)` отрабатывает, `score`/память пополнились; `idle`-агента
  матчинг не трогает. Доказывает шов matching→game **до** оркестратора.
- **DoD:** `pytest tests/matchmaking` зелёный; шов до Игры подтверждён.

## 7. Чистые швы под пост-MVP

- **`scheduled`:** `setup` предпосчитывает полное расписание по стартовым ID
  (round-robin / Berger); `plan_round` отдаёт срез по `round`. Без правок оркестратора.
- **`choice` / `consent`:** интерактивные — через `actor` опрашивают агентов (фаза
  `CHOOSE_PARTNER`/`CONSENT`, расширение `PhaseKind`, mvp-arch §8); пишут `events`.
- **Равная ротация `idle`:** заменить «случайный последний» на счётчик отсидок
  (состояние уже есть — живёт в матчинге между раундами через `setup`).
- **Собственный `rng` матчинга:** производный от seed → план воспроизводим независимо
  от gather-решения в играх (§8.1).

## 8. Решения

1. **`rng` матчинга — РЕШЕНО (2026-06-04): собственный производный `rng`**
   (`Random(f"{seed}:matchmaker")`), не общий с играми. Последовательность разбиений
   по парам воспроизводима по `seed` независимо от игр и планировщика. Это
   единственная воспроизводимость, которую мы хотим (см. 2).
2. **Детерминизм партий под `gather` — СНЯТ (2026-06-04): не нужен.** LLM
   недетерминирован (`temperature>0`), воспроизводимость *внутри* партии не
   ожидается. Воспроизводим только структуру эксперимента (кто с кем) — её даёт п.1.
   Игры берут свой `rng`/недетерминизм как есть.
3. **Сбой агента:** матчинга в `random` не касается (он не зовёт LLM); для `choice` —
   пост-MVP.
